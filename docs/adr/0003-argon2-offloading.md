# 0003. argon2 в пуле потоков и вне транзакции БД

## Status
Accepted

## Context
Нагрузочное тестирование auth-эндпоинтов (register/login) выявило обвал: ~20 RPS,
таймауты 60с под 300 параллельных клиентов. Разбор показал две причины.

1. **Блокировка event loop.** Хеширование пароля argon2 (`recommended()` ≈ 85 мс на хеш,
   67 мс на verify) вызывалось синхронно прямо в async-ручке. На время вычисления
   вставал весь воркер — даже не связанные с auth запросы.
2. **Удержание соединения пула.** Хеш считался **внутри** транзакции UoW:
   ```python
   async with uow:
       user = await uow.users.add(User(hashed_password=hash_password(...)))
   ```
   CPU-bound операция держала соединение PgBouncer все 85 мс. Под нагрузкой пул мгновенно
   исчерпывался → каскад таймаутов.

argon2 медленный **намеренно** (стойкость к перебору) — «ускорить» его нельзя, можно
лишь правильно встроить.

## Decision
1. Выносим argon2 в пул потоков: `hash_password_async`/`verify_password_async` через
   `anyio.to_thread.run_sync`. argon2-cffi отпускает GIL → потоки реально параллелятся
   по ядрам, event loop остаётся свободен.
2. Считаем хеш/verify **вне** транзакции БД: в `register` — до `async with uow`; в
   `login` — берём снимок пользователя, закрываем транзакцию, затем verify.

Тот же принцип «не держать соединение БД во время медленного I/O» применён в
outbox-релее ([0002](0002-transactional-outbox.md)).

## Consequences
- (+) После фиксов и изоляции генератора нагрузки: 0% ошибок, на 4 ядрах ~260 RPS
  смешанного auth-трафика и ~970 RPS на read-пути.
- (+) Тяжёлый CPU больше не блокирует event loop и не пинит соединения пула.
- (−) Пропускная способность самого auth ограничена стоимостью argon2 (≈ ядра / 85 мс);
  масштабируется ядрами/репликами, защищается rate-лимитом — это by design.
- Правило на будущее: любую CPU-bound или сетевую операцию не выполнять внутри открытой
  транзакции UoW.

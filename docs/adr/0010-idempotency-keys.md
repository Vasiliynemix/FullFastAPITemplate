# 0010. Идемпотентность небезопасных операций через Idempotency-Key

## Status
Accepted

## Context
В высоконагруженных системах клиент/сеть ретраят POST-запросы (таймаут, разрыв,
балансировщик). Для денежных операций (`deposit`/`withdraw`) и создания ресурсов это
означает риск **задвоенной** операции: два списания, два счёта. Естественный уникальный
ключ есть не везде (у депозита его нет вовсе), поэтому полагаться на уникальный индекс БД
нельзя. Нужен общий механизм, который делает повтор безопасным и не размазывает логику по
ручкам.

Дополнительно всплыл нюанс контракта ответов: повтор должен вернуть **тот же** ответ, что и
оригинал, — а конверты у нас собираются только через хелперы `success()`/`error()` (см.
[ADR-0001] и `tests/test_response_contract.py`).

## Decision
1. **Заголовок `Idempotency-Key`** (uuid, генерит клиент — один на логическую операцию).
   Стор на Redis (`app/idempotency/store.py`): `try_acquire` ставит `SET NX` маркер
   in-progress, `save_response` сохраняет итоговый ответ под ключом с TTL ~ сутки.
2. **Один хелпер `idempotent(key, model, produce)`** (`app/idempotency/runner.py`) — вся
   лог() в одном месте, в ручке 3 строки:
   * ключа нет → просто `produce()`;
   * ключ с готовым ответом → **replay** из кэша, без повторного выполнения;
   * ключ занят in-flight (параллельный запрос) → `409 Conflict`;
   * иначе → `produce()`, сохранить ответ; при ошибке **отпустить ключ** (ретрай возможен).
3. **Replay восстанавливает ответ через `model.model_validate(cached)`, а НЕ через
   `success()`** — это рехидрация уже собранного конверта (вместе с его `meta`), а не сборка
   нового; `success()` переобернул бы данные. Это единственное легальное исключение из
   контракта «собирай только хелпером», и оно осознанно разрешено тест-гардом.
4. **Обязательность — по риску, на уровне типа-зависимости** (`app/api/deps.py`):
   * `RequiredIdempotencyKey` (без дефолта → `422` без заголовка) — на денежных/создающих
     счёт ручках: `POST /accounts`, `deposit`, `withdraw`;
   * `IdempotencyKey` (опциональный) — на `POST /users` (дубль менее критичен, у user ещё и
     email уникален).
   * Оба валидируют длину ключа `8..255` (отсекает пустую/мусорную строку).

## Consequences
- (+) Денежные ретраи безопасны: повтор по ключу не задваивает операцию, параллельный
  in-flight отбивается `409`. Проверено вживую на проде (replay вернул тот же id; без ключа
  на обязательной ручке — `422`).
- (+) Новой ручке идемпотентность добавляется тремя строками (`idempotent(...)`), исключение
  из контракта ответов инкапсулировано в хелпере, а не размазано.
- (+) Обязательность выражена типом параметра — видна в OpenAPI (`required: true`) и
  проверяется до бизнес-логики, без ручных проверок в теле.
- (−) Клиенты обязательных ручек ДОЛЖНЫ слать заголовок — это ломающее требование для
  существующих интеграций (для template приемлемо; граница «обязателен/опционален»
  настраивается выбором типа-зависимости).
- (−) При replay `meta.request_id` в теле — из исходного запроса (вмёрз в кэш), а заголовок
  `X-Request-ID` ответа — текущего ретрая: они расходятся. Это осознанный выбор («вернуть
  оригинальный ответ дословно» честнее, чем подмешивать новый id в старый ответ).
- (−) Состояние идемпотентности живёт в Redis с TTL: после истечения тот же ключ выполнится
  заново. TTL (~сутки) подобран под реалистичное окно ретраев.

# 0004. Опциональные компоненты: брокер и объектное хранилище за флагами

## Status
Accepted

## Context
Не каждому сервису на базе шаблона нужны все подсистемы. События (брокер) и работа с
файлами (объектное хранилище) — частые, но не обязательные возможности. Грузить их
всегда — это лишние зависимости на старте, лишние подключения и ручки, которые только
расширяют поверхность атаки и сбивают с толку.

Отдельно по хранилищу: изначально было две реализации (S3 и локальная ФС). Локальная
ФС не работает с несколькими репликами (файл на диске одного инстанса не виден другим),
то есть для прода она всё равно непригодна, а поддержка двух бэкендов усложняла шаблон.

## Decision
1. **Брокер и хранилище — за флагами** `BROKER_ENABLED` (по умолчанию `true`) и
   `STORAGE_ENABLED` (по умолчанию `false`). При выключении компонент не подключается в
   lifespan/воркере, а его ручки (`/notifications`, `/files`) не регистрируются —
   роутер собирается функцией `build_api_router()` в `create_app()` по текущим флагам.
2. **Объектное хранилище — только S3-совместимое** (AWS S3 / MinIO / Yandex). Локальная
   ФС-реализация удалена. Абстракция `AbstractStorage` сохранена — другой бэкенд при
   необходимости добавляется одним классом.
3. **PostgreSQL и Redis флагов отключения НЕ получают**: на них завязаны users/auth/
   cache/rate-limit/sessions/idempotency — это фундамент, делать его опциональным значило
   бы каскадно усложнять весь код ради крайне редкого сценария.

## Consequences
- (+) По умолчанию шаблон не тянет S3; брокер гасится одним флагом — «не грузим лишнее».
- (+) Состав ручек предсказуем и зависит от настроек (видно в Swagger).
- (+) Одна реализация хранилища — меньше кода и развилок.
- (−) Для локальной разработки с файлами нужен MinIO (S3 локально), а не просто диск.
- (−) `BROKER_ENABLED=false` при включённом outbox => события копятся (релей не работает);
  в этом случае выключайте и `OUTBOX_ENABLED`.

# 0008. UnitOfWork: ленивые репозитории, один экземпляр на транзакцию

## Status
Accepted

## Context
UnitOfWork в `__aenter__` создавал ВСЕ репозитории сразу и перечислял каждый явно. С ростом
числа репозиториев это: (1) связывает UoW со всеми репо (импорт каждого), (2) требует
правок `__aenter__` на каждый новый репо. Возник вопрос про ленивую загрузку.

Важный контекст про жизненный цикл: UoW — НЕ синглтон. `get_uow()` создаёт новый
`UnitOfWork` на каждый запрос; сервис тоже per-request. Но внутри сервиса `self.uow` один
на все методы — то есть один экземпляр UoW **переиспользуется последовательно** между
разными `async with`.

## Decision
1. **Ленивые репозитории + дженерик-доступ.** Репозиторий создаётся при первом обращении и
   кэшируется на время транзакции. Частые — именованные property (`uow.users`, эргономика +
   типы); любой — через `uow.repo(SomeRepository)` (новый репо НЕ требует правок UoW).
2. **Кэш сбрасывается на каждый вход/выход** (`__aenter__`/`__aexit__`). Это обязательно
   из-за переиспользования: иначе репо повисли бы на старой, уже закрытой сессии.
3. **Контракт «один UoW = одна транзакция за раз» — enforced.** Повторный/вложенный вход на
   активном инстансе бросает RuntimeError (раньше тихо затирал `_session`/`_repos`).
   Для конкурентных транзакций — отдельные UoW (своя сессия у каждого); `AsyncSession` и сама
   не допускает конкурентного доступа.

## Consequences
- (+) UoW не связан со всеми репо; новый репозиторий не требует правок UoW (`uow.repo(X)`).
- (+) Эргономика сохранена: `uow.users`/`uow.accounts` работают как раньше (теперь property).
- (+) Misuse (параллельный/вложенный вход на одном UoW) падает громко, а не плавающим багом.
- (−) Перфа это НЕ ускоряет (создание репо — микросекунды на фоне соединения/запросов);
  ценность — чистота и масштабируемость, а не скорость.
- Замечание: для настоящей конкурентности (как в тестах на гонки) создаются N независимых
  UoW — это норма, а не обход ограничения.

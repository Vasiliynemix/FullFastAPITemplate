# 0005. Защита от гонок: пессимистичная и оптимистичная блокировки

## Status
Accepted

## Context
Unit of Work даёт атомарность (всё-или-ничего в одной транзакции), но НЕ защищает от
**lost update** в read-modify-write при дефолтном уровне изоляции `READ COMMITTED`:

```python
async with uow:
    user = await uow.users.get(id)   # двое читают одну версию
    user.x = ...                      # оба меняют
    await uow.commit()                # last-write-wins -> одно изменение потеряно
```

До этого решения в слое репозитория/UoW не было никаких примитивов управления
конкурентным доступом.

## Decision
Добавляем ОБА классических механизма — разработчик выбирает под характер нагрузки.

1. **Пессимистичная блокировка** — флаги на геттерах `BaseRepository`:
   `get/get_by/list(..., for_update=True)` → `SELECT ... FOR UPDATE`. Плюс `skip_locked`
   (очереди: забрать пачку, минуя занятые строки) и `nowait` (сразу падать, если занято).
   Работает только внутри транзакции UoW; лок держится до commit/rollback.

2. **Оптимистичная блокировка** — `VersionedMixin` (колонка `version_id` + `version_id_col`).
   ORM на каждый UPDATE добавляет `WHERE version_id = <прочитанная>` и инкрементит её;
   проигравший гонку получает `StaleDataError`. Применён к `User`; `UserService.update`
   ловит ошибку и отдаёт **409 Conflict**.

Когда что:
| | `for_update` | `VersionedMixin` |
|---|---|---|
| Держит лок | да (другие ждут) | нет |
| Хорошо при | частых конфликтах (горячая строка) | редких конфликтах |
| Цена конфликта | ожидание в очереди | повтор операции (409) |

## Consequences
- (+) Lost update закрыт двумя способами; проверено: 30 параллельных PATCH на проде дали
  7 успехов + 12 × 409, `version_id` сошёлся (1→8) — потерь нет.
- (+) Оба примитива переиспользуемы (базовый репозиторий / миксин), не привязаны к домену.
- (−) `for_update` нельзя держать во время медленного I/O (сериализует доступ к строке) —
  тот же принцип, что в [0002](0002-transactional-outbox.md)/[0003](0003-argon2-offloading.md).
- (−) Bulk-апдейты Core (`update_by_id`) НЕ проверяют версию — by design (версия только для
  ORM-апдейтов load→mutate→commit).
- Не реализовано (осознанно): клиентский ETag/If-Match (ловить конфликт через долгую правку
  формы, а не только серверную гонку) и `SERIALIZABLE` + авто-ретрай — добавляется при нужде.

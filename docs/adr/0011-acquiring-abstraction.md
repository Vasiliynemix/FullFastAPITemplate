# 0011. Абстракция платёжного эквайринга (SDK vs свой HTTP-клиент)

## Status
Accepted

## Context
Нужен приём платежей. Провайдеров много (YooKassa, Stripe, …), у одних есть собственный
SDK, у других — только REST. Не хочется завязывать бизнес-логику на конкретного провайдера
и его формат ответа, но и нельзя терять доступ к провайдер-специфичным полям. Плюс часть
SDK синхронные (requests) — нельзя звать их прямо из async-обработчика (заблокируют loop).

## Decision
1. **Абстракция `AbstractAcquirer`** ([app/acquiring/base.py]) по аналогии с
   `AbstractBroker`/`AbstractStorage`: домен работает с моделями `Payment`/`Refund`/
   `WebhookEvent`, методы `create/get/capture/cancel/refund/parse_webhook`.
2. **Модели — дженерики по `RawT`** (тип «сырого» ответа в поле `raw`). Наружу всегда
   единые доменные модели (обращение через мой интерфейс), а `raw` оставляет доступ к
   нативному ответу провайдера для специфики.
3. **Правило транспорта:**
   * провайдер С SDK → используем РОДНОЙ клиент SDK (не свой HTTP). Синхронный SDK уносим в
     пул потоков через `anyio.to_thread` (тот же приём, что argon2, [ADR-0003]). `RawT` =
     объект SDK. Пакет SDK импортируется ЛЕНИВО (как aiokafka/aioboto3 у брокера/хранилища) —
     не тянется, если эквайринг не используется.
   * провайдер БЕЗ SDK → клиент на `BaseHTTPClient` (пул, ретраи, маппинг ошибок, см.
     app/clients), `RawT` = `dict`.
4. **Ошибки** провайдера → `AcquiringError` (наследник `ServerException` 502) либо
   `NotFoundError` (платёж не найден) — улетают в единый envelope без сырых трейсбеков.
5. **Реестр вместо «одного типа»** (в отличие от брокера/хранилища): платёжных систем может
   работать НЕСКОЛЬКО одновременно, поэтому у каждой свой флаг (`YOOKASSA_ENABLED`, …), а
   фабрика собирает все включённые в реестр, ключ — **enum `AcquirerName`** (не строки):
   `get_acquirer(AcquirerName.YOOKASSA)`, опечатка ловится типами. Конфиг валидируется на
   старте (`memory` в проде, включённый провайдер без кред → ошибка при запуске). Добавить
   провайдера = значение в `AcquirerName` + флаг/креды + ветка в фабрике.
6. **Реализации**: `YooKassaAcquirer` (SDK) и `InMemoryAcquirer` (заглушка без сети,
   `RawT=dict`) — для тестов/локалки. memory **запрещён в проде** валидатором конфига
   (старт падает явно), чтобы не принимать «игрушечные» платежи в проде.
7. **Вебхук** — `parse_webhook` только парсит тело; достоверность подтверждается
   перепроверкой статуса через `get_payment` (уведомление можно подделать, YooKassa его не
   подписывает).

## Consequences
- (+) Смена провайдера не трогает бизнес-логику; провайдер-специфика доступна через `raw`.
- (+) Один контракт под «есть SDK» и «нет SDK» — каждый использует оптимальный транспорт.
- (+) Синхронный SDK не блокирует event loop (offload в пул потоков).
- (+) Несколько платёжек одновременно (реестр по флагам) — добавление новой не трогает
  существующие; легко A/B/фолбэк между провайдерами.
- (+) Тестируемо без сети/кредов (`InMemoryAcquirer`) — маппинг YooKassa проверяется на
  фейковом объекте SDK, флоу — на заглушке; memory в проде запрещён валидатором.
- (−) `RawT` у SDK-реализации типизирован как `Any` (стабов SDK нет) — статика не видит полей
  нативного объекта; маппинг защищён `getattr` и покрыт тестом.
- (−) Двухстадийные платежи/частичные возвраты providers реализуют по-разному — абстракция
  даёт общий минимум; экзотику тащите через `raw` или доп. методы реализации.
- (−) Компонент пока не внедрён в роуты/lifespan (намеренно) — интеграцию и обработку
  вебхуков делает потребитель шаблона под свой домен.

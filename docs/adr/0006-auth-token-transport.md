# 0006. Транспорт токена авторизации: header XOR cookie

## Status
Accepted

## Context
Изначально токены отдавались только в теле ответа, а авторизация читала их только из
заголовка `Authorization: Bearer`. Появилась потребность в браузерном сценарии — токен в
куке, недоступной JS. Возник вопрос: поддерживать оба способа одновременно или выбирать?

Два способа имеют противоположные профили безопасности:
* **header (Bearer)** — не отправляется браузером автоматически => иммунен к CSRF, но токен
  хранится в JS (localStorage) => уязвим к краже через XSS;
* **cookie (HttpOnly)** — недоступна JS => иммунна к краже токена через XSS, но
  отправляется автоматически => нужна защита от CSRF (SameSite).

В проде выбирают ОДИН способ под тип клиента (SPA/mobile → header, браузерная сессия →
cookie). Поддержка обоих сразу — лишняя поверхность и неоднозначность «что главнее».

## Decision
Один флаг-селектор `AUTH_TOKEN_TRANSPORT = header | cookie` (по умолчанию `header`) —
**что-то одно**:
* `header`: токен в `Authorization`; куки не ставятся; авторизация читает только заголовок;
* `cookie`: на login/refresh ставятся HttpOnly-куки (access — Path `/`, refresh — Path
  `/api/v1/auth`), logout их чистит; авторизация читает токен только из куки.

Запрещённые комбинации падают на старте (fail-fast валидатор, а не тихое переключение):
* `cookie` требует `AUTH_JWT_ENABLED=true` (без JWT токена нет);
* `cookie` несовместим с `GLOBAL_API_KEY_ENABLED=true` — браузеру негде безопасно хранить
  секретный глобальный ключ.

## Consequences
- (+) Явный выбор стратегии под клиента; нет двусмысленности «оба сразу».
- (+) Неверный конфиг ловится на старте с понятной ошибкой, а не молча в проде.
- (+) HttpOnly+Secure+SameSite=lax по умолчанию — токен недоступен JS, базовая защита от CSRF.
- (−) Cookie-режим и глобальный API-ключ взаимоисключающи — это осознанное ограничение
  (они для разных аудиторий: браузер vs service-to-service).
- (−) `SameSite=lax` не покрывает кросс-доменный фронт — там нужен `none`+Secure+CSRF-токены
  (не реализовано; задокументировано как расширение).

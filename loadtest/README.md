# Нагрузочное / дымовое тестирование (k6)

Скрипт [`k6.js`](k6.js) гоняется через официальный образ `grafana/k6` в Docker —
ставить ничего не нужно. Контейнер k6 подключается к сети prod-стека
(`fastapi_default`) и стучится в nginx по внутреннему DNS-имени `nginx`.

## Быстрый старт

```bash
make prod-up          # стек должен быть поднят
make loadtest         # smoke + traffic + ratelimit через nginx
```

## Режимы (`mode=`)

| Режим | Что делает |
|---|---|
| `smoke` | по одному разу дёргает **каждую** ручку (вкл. admin-delete, refresh, logout), проверяет статусы. Порог: 99% проверок зелёные |
| `traffic` | объёмная смешанная нагрузка с ramp-up: register→login→CRUD→пагинация→стрим→уведомления |
| `ratelimit` | долбит из **одного** IP (80 rps) → доказывает app-лимит, считает `app_rate_limited_429` |
| `all` | всё по очереди (по умолчанию) |

## Параметры

| Параметр | Дефолт | Смысл |
|---|---|---|
| `mode=` | `all` | см. таблицу выше |
| `vus=` | `50` | виртуальных пользователей в `traffic` |
| `dur=` | `30` | длительность `traffic`, **секунды** |
| `url=` | `https://nginx` | базовый URL. `http://backend:8080` — мимо nginx |
| `spread=` | `true` | уникальный `X-Forwarded-For` на VU (обходит app-лимит); `false` — общий |
| `api_key=` | пусто | `X-API-Key`, если включён `GLOBAL_API_KEY_ENABLED` |

## Примеры

```bash
# Просто проверить, что все ручки живые
make loadtest mode=smoke

# Серьёзная нагрузка: 200 VU, минута
make loadtest mode=traffic vus=200 dur=60

# Доказать, что rate limit работает (увидишь app_rate_limited_429 > 0)
make loadtest mode=ratelimit

# Настоящая ёмкость бэкенда МИМО nginx (nginx режет на 200 r/s по IP контейнера)
make loadtest url=http://backend:8080 mode=traffic vus=300 dur=60
```

## Как читать результат

k6 в конце печатает сводку. На что смотреть:

- **`checks`** — доля успешных проверок. В `smoke` должно быть `100%`.
- **`http_req_duration`** — латентность; `p(95)` — «медленные» 5%.
- **`http_reqs`** — суммарный RPS (значение `.../s`).
- **`app_rate_limited_429`** — сколько запросов срезал app-лимитер (ждём >0 в `ratelimit`).
- **`nginx_throttled_503`** — сколько срезал глобальный лимит nginx (200 r/s на IP контейнера; растёт, если льёшь через nginx выше потолка).

## Важно про лимиты (почему два «потолка»)

- **nginx** лимитит по реальному TCP-IP (`$binary_remote_addr`) → **200 r/s + burst 400** на
  IP контейнера k6. Через `https://nginx` весь тест упрётся в это; превышение → `503`.
- **приложение** лимитит по `X-Forwarded-For` → **burst 20/с** и **1000/60с** на «клиента».
  `spread=true` разводит клиентов по разным бакетам (обход), общий IP в `ratelimit` — ловит 429.

Чтобы померить честную ёмкость бэкенда без nginx-потолка — бей напрямую:
`make loadtest url=http://backend:8080`.
